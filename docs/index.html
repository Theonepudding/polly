Too Tired [T_T]

Member portal and Discord bot for the Too Tired [T_T] Free Company on Spriggan [Chaos], EU. Built on Next.js 15, deployed on Cloudflare Workers.

Member Roster

Lodestone Sync

Member data (name, rank, level, join date, portrait URL) is scraped directly from the FC's Lodestone page. Syncs run automatically every 30 minutes in the background via AutoSync, or can be triggered manually from the Members page or Admin panel.

Join history is preserved in KV so returning members keep their original join date. A join-history map (lodestoneId → isoDate) is maintained alongside the main member list and checked before any date is overwritten.

New Member Detection

Members who joined within the last 30 days automatically receive a teal ✦ NEW ribbon and a glowing animated card border.

Admin override — if automatic detection is wrong (bad join date, returning player, etc.), admins can hover any member card and toggle its new status with the pill button that appears. Stored as manualNew: true | false | null on the member. null means use the automatic 30-day rule.

Portrait Images

Portraits are fetched from Lodestone and served via /api/portrait/[lodestoneId]. Falls back to an initials avatar in the member's rank colour. A bulk refresh can be triggered from the Admin panel.

Filters & Sorting

Search by name, filter by rank, sort by join date / name / level. A "New only" toggle narrows to recently joined members. The default view groups members by rank with a count per group.

Member Profiles

Each member has a dedicated profile page at /members/[id] that aggregates data from multiple external sources in parallel.

Character Data

Pulled from Lodestone via getCharacterData(lodestoneId) and cached in KV under xivapi-char-{lodestoneId} for 1 hour. On member profile pages the cache is warm-only — stale data is shown instantly while a background refresh runs, rather than blocking the page.

  • Job Levels — all combat, crafting and gathering jobs scraped from the character's Lodestone page, grouped by role (Tank, Healer, DPS, Crafter, Gatherer). Jobs at level 100 are highlighted in gold with a glow effect.
  • Mount & Minion counts — total counts shown in the Collections header.

Collections (Mounts & Minions)

Fetched from the character's Lodestone mount/minion pages. Each item requires a separate tooltip fetch to resolve its name — tooltips are stored permanently in KV as a single JSON map (lod-tooltip-map), keyed internally by the 40-character content hash from the Lodestone tooltip URL. Each unique mount or minion is only ever fetched once across all characters, and the entire map is written in a single KV put per scrape.

Items are then enriched with data from the FFXIV Collect API (large render image, global ownership %, acquisition sources). The full mount/minion catalog is cached in KV as ffxiv-collect-catalog-{type} for 24 hours.

Clicking an item opens a modal with the large render, rarity percentage, acquisition sources, and a link to FFXIV Collect.

Notable Feats

Scraped from ffxivcollect.com/characters/{id}/stats/rarity — the top 10 rarest achievements and top 10 rarest titles owned by the character. Cached under ffxiv-feats5-{lodestoneId} for 6 hours.

  • Achievement entries show: icon, name, global ownership %, and on click — how to obtain and any title reward.
  • Title entries show: title text in quotes, the achievement that grants it, and on click — the full "how to obtain" description.

HTML entities in scraped names (e.g. &#39; → ') are decoded before display or caching.

Poll Activity

The last 30 polls are scanned for votes by the member (matched by username). Each found vote shows the poll title, chosen option, and active/closed status — linking through to the full poll.

Community Polls

Creating a Poll

Admins (Dreamscaper+) open the "New Poll" modal and set:

  • Title and optional description
  • Up to 6 options
  • Time-slot voting — voters also pick a preferred time after choosing their option. Presets are 17:00–21:00 UTC, or any custom time. Stored in UTC, displayed in each voter's local timezone.
  • Poll duration: 1, 3, 7 or 14 days

After creation, a prompt asks whether to post to Discord immediately.

Voting

One vote per member. Requires login via Discord OAuth. Results update live with bar charts. Members with Sleep Paralysis rank cannot vote.

Permissions

ActionWho
Create pollAdmins — Dreamscaper, Moonwarden, The Oversleeper
VoteAll members except Sleep Paralysis
Close a pollAdmins and the poll creator
Delete a pollAdmins only — prompts a custom confirmation modal
Send / resend to DiscordAdmins only

All destructive actions use a styled custom confirmation modal with the item name in the message. No native browser confirm() dialogs anywhere in the site.

Scheduled / Recurring Polls

Admins create poll templates that automatically post a fresh poll on a repeating schedule. Managed under the Scheduled tab on the Polls page (admin-only tab).

Template Configuration

  • Title, description, options — same as a one-off poll
  • Time-slot voting — optional, identical setup to one-off polls
  • Repeat interval — Daily / Weekly / Bi-weekly / Monthly presets, or any custom number of days (1–365)
  • First poll date — a date picker lets you choose the exact start. For weekly schedules, the chosen weekday becomes the recurring day (pick a Monday → fires every Monday)
  • Post time — quick preset chips shown in your local time, plus a full custom time input. Stored internally as UTC hour.
  • Poll duration — how many days each generated poll stays open
  • Post to Discord — whether each new poll is auto-announced

Template Actions

  • ▶ Run now — fires the template immediately, ignoring the schedule
  • ▮▮ Pause / Resume — suspends or re-activates without deleting
  • 🗑 Delete — removes the schedule permanently. Polls already created are kept.

Drift-free Scheduling

Next run time is always computed from the previous scheduled time, not wall-clock time. If a cron fires 30 minutes late, the next slot is still exactly intervalDays after the original intended time — no slow drift over weeks.

Discord Integration

Bot Setup (one-time)

Visit /bot-admin. Only users whose Discord IDs are in the bot-admins KV key can access this panel.

  1. Click Invite Bot — opens the OAuth invite URL for the bot
  2. Select the server from the dropdown (lists servers the bot is already in)
  3. Select the announcement channel where polls will be posted
  4. Click Save Setup

Poll Embeds

Polls posted to Discord appear as rich embeds with:

  • Title and description
  • Vote count and closing date
  • A dynamically generated image card showing live results (see Text-to-Image Cards)
  • Vote buttons, one per option (up to 5), in Discord Blurple / Primary style

Smart Resend

The "Send to Discord" admin button first tries to update the existing Discord message. If it gets a 404 (message deleted or in a different channel), it automatically creates a new post and saves the new message ID. No manual intervention needed.

Closed Poll Embeds

When a poll closes (manually or by expiry), the Discord embed updates: buttons are removed, the card image switches to teal CLOSED styling, and bars show the final tallies.

Text-to-Image Cards

Poll results are shown in Discord as a dynamically generated PNG image. There is no static screenshot — the image is rendered fresh on every request using server-side JSX processed by satori (the engine inside next/og). The text, bars and voter names are all drawn into the image at render time.

What the card contains

  • Poll title with a mini bar-chart icon (gold when open, teal when closed)
  • OPEN or CLOSED status badge
  • Each option with a percentage fill bar
  • Vote count per option
  • Voter names below each bar — up to 6, then "+N more"
  • Total vote count and closing date in the footer

Resolution & sharpness

Cards are generated at 1200 × (dynamic height) px — 2× the display size. Discord scales embed images down to roughly 600 px wide, so the 2× source stays sharp at any display density. All measurements in the satori template are doubled to match.

Cache busting

The image URL in each Discord message includes ?t=<timestamp>, updated every time the embed refreshes. Discord uses the full URL as its image cache key, so it always re-fetches the latest result image rather than showing stale bars.

Image route

GET /api/poll-image/[pollId]?t=<timestamp>

Returns a PNG. Paste any poll's image URL in a browser to preview the card. Intentionally public and unauthenticated — Discord must fetch it without credentials.

Site OG Image (Link Previews)

When the site URL is shared on Discord, Twitter/X, Slack, iMessage, etc., a custom-generated 1200 × 630 px preview card appears instead of a generic screenshot. Rendered on demand by next/og — not a static file.

Design

  • Dark navy gradient background matching the site's colour scheme
  • Scattered stars in white, gold and teal
  • Gold top edge line + teal bottom edge line spanning the full width
  • Inner panel with a gold border — matching the site's glass-gold card style
  • Bar-chart icon ornament (same icon used on poll cards)
  • "Too Tired" in large white Cinzel bold + "[T_T]" in site gold
  • FINAL FANTASY XIV FREE COMPANY in gold caps
  • SPRIGGAN · CHAOS · EU in dim white

Cinzel font is fetched from Google Fonts at request time. If unavailable, falls back to next/og's bundled Noto Sans.

Busting Discord's cache

Discord caches OG image data per URL for up to 24 hours. To force an immediate refresh, share the URL with any query string — Discord treats it as a new URL and fetches fresh metadata:

https://tired.pudding.vip/?v=2

After a design update to the OG image, bump the version number in src/app/layout.tsx to permanently bust the cache for all future shares:

images: [{ url: '/opengraph-image?v=3', width: 1200, height: 630 }]

Preview without sharing

Open /opengraph-image directly in a browser to see the current card without posting a link anywhere.

Environment Variables

Set in the Cloudflare Pages dashboard under Settings → Environment Variables for production, or in .env.local for local development.

# NextAuth
NEXTAUTH_URL=https://your-domain.com
NEXTAUTH_SECRET=<random 32+ char secret>

# Discord OAuth app (member login)
DISCORD_CLIENT_ID=<your app client id>
DISCORD_CLIENT_SECRET=<your app client secret>

# Discord Bot (posting polls, reading servers/channels)
DISCORD_BOT_TOKEN=<your bot token>

# Cron protection (recommended)
CRON_SECRET=<random secret>

The Cloudflare KV namespace is declared in wrangler.toml as binding FC_KV. No connection string needed — the Worker accesses it natively.

Character & collection caches

These keys are populated on demand and expire by TTL check (not KV expiry), so stale entries are overwritten on next fetch.

Key patternTTLContents

lod-tooltip-mappermanentJSON map of {hash: name} for all resolved mount/minion tooltips. Hash is the 40-char content hash from the Lodestone tooltip URL. Written as a single key (one KV put per scrape) instead of one key per item.

Cron Setup

The scheduled poll runner is a single HTTP endpoint. Call it on any schedule — once per hour is enough since templates manage their own nextRunAt.

POST https://your-domain.com/api/cron/polls
Authorization: Bearer YOUR_CRON_SECRET

Cloudflare Cron Triggers (recommended)

  1. Cloudflare Dashboard → Workers & Pages → your worker
  2. Settings → Triggers → Cron Triggers → Add
  3. Expression: 0 * * * * — top of every hour

External schedulers

  • cron-job.org — free tier is sufficient
  • GitHub Actions with a schedule: trigger
  • Uptime Robot with monitor type HTTP
Security: if CRON_SECRET is not set, the endpoint is unprotected. Always set it in production.

Changing the Domain

Do this first — before deploying. Add the new callback URI to your Discord app in the Discord Developer Portal under OAuth2 → Redirects: https://NEW-DOMAIN/api/auth/callback/discord. Without this, all logins will fail the moment traffic moves to the new domain.

The domain is referenced in 5 places. All must be updated before deploying.

  • wrangler.toml

    Update the routes array. The new zone must already exist in your Cloudflare account.

    routes = [
  { pattern = "new-domain.com", custom_domain = true, zone_name = "new-domain.com" }
]
  • src/app/layout.tsx

    Update metadataBase (controls all absolute OG / canonical URLs). Also bump the v= number on the OG image to bust Discord's embed cache.

    metadataBase: new URL('https://new-domain.com'),

// inside openGraph.images — increment v= to bust Discord cache
{ url: '/opengraph-image?v=3', width: 1200, height: 630 }
  • Cloudflare Pages
    NEXTAUTH_URL env var

    Cloudflare Pages dashboard → Settings → Environment Variables. Set NEXTAUTH_URL to https://new-domain.com. NextAuth uses this to construct OAuth callback URLs. Without this, OAuth logins redirect to the old domain and fail.

  • .env.local

    Update NEXTAUTH_URL if you point local dev at a real domain. Usually kept as http://localhost:3000 for development, so this may not need changing.

  • Discord Developer Portal
    OAuth2 Redirects

    Add https://new-domain.com/api/auth/callback/discord to the redirect URIs list. You can keep the old entry during the transition period and remove it once the switch is confirmed working.

After the switch

  • Test Discord OAuth login end-to-end on the new domain
  • Re-run Bot Admin setup at /bot-admin if the Discord app's interactions endpoint URL needs updating for slash commands
  • Share https://new-domain.com/?v=1 on Discord to verify the new OG image card loads correctly
  • Update the live URL badge at the top of docs/index.html
